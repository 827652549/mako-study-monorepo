## 🔌 连接性能测试

### 网络延迟分析

```
平均延迟：     208.98ms
中位数：       164.93ms

P99：          1017.72ms
最小延迟：     162.89ms
最大延迟：     1017.72ms
```

**延迟分布：**
- <100ms：   0%
- 100-200ms：95% ✅ （大多数请求响应良好）
- 200-300ms：0%
- 300-500ms：0%
- >500ms：   5% ⚠️ （偶发延迟峰值）

**分析：**
- 大部分连接延迟在 160-180ms，这是网络延迟和数据库处理时间的总和
- 延迟中位数约 165ms，这是正常的国际连接延迟
- 存在约 5% 的请求延迟超过 1 秒，可能与数据库连接池管理有关

## 📝 数据操作性能

### 单条插入 vs 批量插入

| 操作类型 | 耗时 | 吞吐量 |
|---------|------|--------|
| 单条插入 | ~172ms | 每秒 ~6 条 |
| 批量插入（10条） | ~338ms | 每秒 ~30 条 |
| 批量插入（50条） | ~360ms | 每秒 ~138 条 |
| 批量插入（100条） | ~367ms | 每秒 ~272 条 |
| 批量插入（500条） | ~357ms | 每秒 ~1,400 条 |

**关键发现：**
- 单条插入效率很低（~172ms/条）
- 批量插入显著提升吞吐量，500条批次可达 **1,400条/秒**
- 单条操作的延迟主要来自网络往返时间

### 数据量对查询性能的影响

```
10 条记录：
  插入：334.01ms
  全查询：170.24ms
  JSONB查询：165.27ms

100 条记录：
  插入：496.89ms
  全查询：325.98ms
  JSONB查询：162.10ms

1,000 条记录：
  插入：700.29ms
  全查询：672.94ms
  JSONB查询：164.39ms

10,000 条记录：
  插入：1,375.56ms
  全查询：1,068.75ms
  JSONB查询：164.47ms
```

**重要发现：**
- 全表查询耗时随数据量线性增长
- JSONB 查询时间基本恒定 (~165ms)，说明有效的索引优化
- 10,000 条记录的全查询耗时约 1.07 秒

## ⚠️ 发现的问题

1. **并发连接限制**
   - 使用连接池 (Pooler) 时有并发限制
   - 建议使用 Supabase 的连接池端点，而非直连

2. **延迟波动**
   - P95 延迟高达 1000ms+
   - 可能是连接初始化或数据库垃圾收集导致

3. **单条操作效率低**
   - 单条插入/更新/删除 172ms 左右
   - 不适合频繁的单条操作

4. **QPS 较低**
   - 5.67 QPS 的连接稳定性测试中比较低
   - 可能受限于网络延迟

## 💡 优化建议

### 1. 使用批量操作 🚀 高优先级

**建议：** 使用批量插入替代单条插入

```javascript
// ❌ 不推荐：单条插入（172ms × 100 = 17,200ms）
for (let i = 0; i < 100; i++) {
  await sql`INSERT INTO users VALUES (...)`
}

// ✅ 推荐：批量插入（500ms，速度快 34 倍）
const values = Array.from({ length: 100 }, (_, i) => [...])
await sql`INSERT INTO users VALUES ${sql(values)}`
```

### 2. 启用连接池 🚀 高优先级

**当前配置：**
```
主机：aws-1-ap-southeast-2.pooler.supabase.com
```

这已经是连接池端点。如需进一步优化，考虑：
- 增加连接池大小（在 Supabase 仪表板配置）
- 使用 PgBouncer 自建连接池

### 3. 创建索引 📋 中等优先级

对于大型表，为常查字段创建索引：
```sql
CREATE INDEX idx_users_email ON users(email);
CREATE INDEX idx_posts_user_id ON posts(user_id);
CREATE INDEX idx_created_at ON users(created_at);
```

### 4. 优化查询 📋 中等优先级

- 使用 LIMIT 限制返回结果数量
- 避免 SELECT *，只选需要的列
- 对大表使用分页查询

```javascript
// ❌ 不推荐（可能返回 10,000 条记录）
const users = await sql`SELECT * FROM users`

// ✅ 推荐（分页查询）
const pageSize = 50
const page = 1
const users = await sql`
  SELECT id, name, email FROM users
  LIMIT ${pageSize}
  OFFSET ${(page - 1) * pageSize}
`
```

### 5. 缓存策略 📋 中等优先级

- 使用 Redis 缓存常查数据
- 减少数据库往返次数
- 特别适合高读取场景

### 6. 地域选择 🌍 可选

- 当前使用 AP Southeast 2（澳大利亚）
- 如用户在中国，可考虑迁移到更近的地区（需要联系 Supabase 支持）
- 但地理距离改变可能只能减少 50-100ms 延迟

### 7. 实时同步 ⚡ 可选

```javascript
// 使用 Supabase Realtime 而非轮询
const subscription = supabase
  .from('users')
  .on('*', payload => {
    console.log('Change received!', payload)
  })
  .subscribe()
```

## 总结

Supabase 在以下方面表现良好：
- ✅ **连接稳定性**：100% 成功率
- ✅ **批量操作**：1,400+ 条/秒
- ✅ **功能完整性**：支持 JSONB、事务等

需要改进的方面：
- ⚠️ **延迟**：平均 165-270ms（受地理位置影响）
- ⚠️ **单条操作效率**：~172ms（不适合高频操作）
- ⚠️ **QPS**：5.67（高并发场景需优化）

**建议：**
- 优先使用批量操作
- 启用连接池
- 为大表创建索引
- 考虑加入缓存层（Redis）
- 如延迟成为问题，考虑迁移地域或使用自建数据库

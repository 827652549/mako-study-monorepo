# 任务列表：[FEATURE NAME]

**输入**：来自 `/specs/[###-feature-name]/` 的设计文档
**前置条件**：plan.md（必须）、spec.md（用户故事必须）、research.md、data-model.md、contracts/

**测试**：以下示例包含测试任务。测试是可选的——仅在功能规格中明确要求时才包含。

**组织方式**：任务按用户故事分组，以便独立实现和测试每个故事。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**：可并行执行（涉及不同文件，无依赖关系）
- **[Story]**：该任务所属的用户故事（如 US1、US2、US3）
- 描述中包含精确的文件路径

## 路径约定

- **单一项目**：仓库根目录下的 `src/`、`tests/`
- **Web 应用**：`backend/src/`、`frontend/src/`
- **移动端**：`api/src/`、`ios/src/` 或 `android/src/`
- 以下路径基于单一项目结构——请根据 plan.md 的实际结构进行调整

## 依赖关系与执行顺序

### 阶段依赖关系

- **初始化（第一阶段）**：无依赖——可立即开始
- **基础建设（第二阶段）**：依赖初始化完成——阻塞所有用户故事
- **用户故事（第三阶段及以后）**：均依赖基础建设阶段完成
  - 用户故事可并行推进（如有足够人力）
  - 或按优先级顺序依次推进（P1 → P2 → P3）
- **打磨（最终阶段）**：依赖所有目标用户故事全部完成

### 用户故事间依赖关系

- **用户故事 1（P1）**：基础建设（第二阶段）完成后即可开始——不依赖其他故事
- **用户故事 2（P2）**：基础建设（第二阶段）完成后即可开始——可与 US1 集成，但应可独立测试
- **用户故事 3（P3）**：基础建设（第二阶段）完成后即可开始——可与 US1/US2 集成，但应可独立测试

### 每个用户故事内部顺序

- 测试（如包含）必须先编写，并确保在实现前处于失败状态
- 模型先于服务
- 服务先于接口
- 核心实现先于集成
- 本故事完成后再进入下一优先级

### 并行机会

- 所有标记 [P] 的初始化任务可并行执行
- 所有标记 [P] 的基础建设任务可在第二阶段内并行执行
- 基础建设阶段完成后，所有用户故事可并行启动（如团队人力允许）
- 某用户故事中所有标记 [P] 的测试任务可并行执行
- 某故事中标记 [P] 的模型任务可并行执行
- 不同用户故事可由不同团队成员并行推进

## 实现策略

### MVP 优先（仅用户故事 1）

1. 完成第一阶段：初始化
2. 完成第二阶段：基础建设（关键——阻塞所有故事）
3. 完成第三阶段：用户故事 1
4. **暂停并验证**：独立测试用户故事 1
5. 如已就绪则部署/演示

### 增量交付

1. 完成初始化 + 基础建设 → 基础设施就绪
2. 添加用户故事 1 → 独立测试 → 部署/演示（MVP！）
3. 添加用户故事 2 → 独立测试 → 部署/演示
4. 添加用户故事 3 → 独立测试 → 部署/演示
5. 每个故事在不破坏前序故事的情况下持续增加价值

### 多人并行策略

多名开发者协作时：

1. 团队共同完成初始化 + 基础建设
2. 基础建设完成后：
   - 开发者 A：用户故事 1
   - 开发者 B：用户故事 2
   - 开发者 C：用户故事 3
3. 各故事独立完成并集成

## 备注

- [P] 任务 = 涉及不同文件，无依赖关系
- [Story] 标签将任务与特定用户故事关联，便于追溯
- 每个用户故事应可独立完成并测试
- 实现前请验证测试处于失败状态
- 每完成一个任务或逻辑组后提交代码
- 可在任意检查点暂停，独立验证故事是否完整
- 避免：任务描述模糊、同一文件冲突、破坏独立性的跨故事依赖

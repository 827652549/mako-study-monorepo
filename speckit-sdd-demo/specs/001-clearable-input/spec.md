# 功能规格：带清除按钮的输入框组件

**功能分支**：`001-clearable-input`
**创建日期**：2026-04-16
**状态**：草稿
**输入**：用户描述："我要做一个带清楚按钮的输入框组件"

## 用户场景与验收 *(必填)*

### 用户故事 1 - 输入内容后一键清除（优先级：P1）

用户在输入框中输入内容后，右侧出现一个清除按钮（×），点击后输入框内容被清空，光标重新聚焦到输入框。

**为什么是这个优先级**：这是组件的核心价值——让用户无需逐字删除即可快速清空内容，是最基本的 MVP 能力。

**独立验收方式**：在页面上放置该组件，输入任意文字后点击清除按钮，验证内容清空且光标聚焦。

**验收场景**：

1. **假设** 输入框为空，**当** 用户查看输入框，**则** 清除按钮不显示
2. **假设** 输入框有内容，**当** 用户查看输入框，**则** 清除按钮显示在右侧
3. **假设** 输入框有内容且清除按钮可见，**当** 用户点击清除按钮，**则** 输入框内容清空，清除按钮消失，输入框获得焦点

---

### 用户故事 2 - 支持外部受控使用（优先级：P2）

开发者可以将该组件作为受控组件使用，传入 `value` 和 `onChange`，清除操作触发 `onChange('')`，组件状态由外部管理。

**为什么是这个优先级**：实际项目中表单通常是受控的，支持受控模式才能被广泛集成。

**独立验收方式**：在父组件中绑定 state，点击清除后验证父组件 state 变为空字符串。

**验收场景**：

1. **假设** 组件以受控方式使用（传入 value），**当** 用户点击清除按钮，**则** 触发 onChange 回调且参数为空字符串
2. **假设** 外部 value 变为空，**当** 组件重新渲染，**则** 清除按钮不显示

---

### 用户故事 3 - 支持禁用状态（优先级：P3）

当组件被禁用时，输入框不可编辑，清除按钮不显示或不可点击。

**为什么是这个优先级**：表单禁用状态是常见需求，但不影响核心功能，优先级低于受控模式。

**独立验收方式**：传入 disabled 属性，验证输入框无法输入且清除按钮不可交互。

**验收场景**：

1. **假设** 组件设置了禁用状态，**当** 用户查看组件，**则** 输入框不可编辑，清除按钮不显示

---

### 边界情况

- 输入框内容全为空格时，是否显示清除按钮？（假设：显示，空格也算有内容）
- 用户用键盘（Escape 键）能否触发清除？（假设：v1 不支持，仅鼠标点击）
- 组件是否需要支持 placeholder？（假设：支持，透传给内部输入框）

## 需求 *(必填)*

### 功能需求

- **FR-001**：输入框内容为空时，清除按钮必须不可见
- **FR-002**：输入框有内容时，清除按钮必须显示在输入框右侧内部
- **FR-003**：用户点击清除按钮后，输入内容必须被清空
- **FR-004**：清除后，输入框必须自动获得焦点
- **FR-005**：组件必须支持受控模式（外部传入 value + onChange）
- **FR-006**：组件必须支持非受控模式（内部维护状态）
- **FR-007**：组件被禁用时，输入框不可编辑，清除按钮不显示
- **FR-008**：组件必须支持 placeholder 属性

## 成功标准 *(必填)*

### 可量化指标

- **SC-001**：用户从「想清空」到「完成清空」只需 1 次点击，无需多次操作
- **SC-002**：清除按钮的出现/消失与输入内容变化同步，无明显延迟
- **SC-003**：组件可无缝替换现有普通输入框，不破坏原有表单逻辑

## 假设与约束

- 用户使用现代浏览器（不需要兼容 IE）
- v1 仅支持单行文本输入，不涉及多行（textarea）
- 清除按钮样式使用简单的「×」文本或图标，不依赖图标库
- 组件不包含校验逻辑，校验由外部表单框架处理
